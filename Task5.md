| Встреча | Частота	| Участники | Цель |
| -------- |-------- |---------- | ------------ |
| Daily Standup (15 мин) | Ежедневно | Вся команда | Синхронизация по прогрессу за день: что сделано, что планируется, какие блокеры. Выявляет зависимости между разработчиками до того, как они становятся задержками. |
| -------- |-------- |---------- | ------------ |
| Sprint Planning (2–3 часа) | Раз в 2 нед. | Вся команда | Выбор задач из бэклога в спринт, декомпозиция и оценка в story points. Команда берёт обязательство по scope — это основа предсказуемости для стейкхолдеров. |
| -------- |-------- |---------- | ------------ |
| Sprint Review (1 час) | Раз в 2 нед. | Команда + стейкхолдеры | Демонстрация рабочего инкремента первым B2B-клиентам или инвесторам. Сбор конкретной обратной связи по готовым фичам — источник для следующего планирования. |
| -------- |-------- |---------- | ------------ |
| Retrospective (1 час) | Раз в 2 нед. | Вся команда | Анализ процессов внутри команды: что работало хорошо, что мешало, какие изменения внедрить в следующий спринт. Обязательная встреча — команда MVP должна учиться быстрее, чем конкуренты. |
| -------- |-------- |---------- | ------------ |
| Backlog Refinement (1 час) | Еженедельно | PM + разработчики | Детализация и оценка задач следующих спринтов. Исключает ситуацию, когда на planning приходят сырые истории без критериев приёмки — это главный источник задержек в малых командах. |
| -------- |-------- |---------- | ------------ |
| Product Sync (45 мин) | Еженедельно | PM + UX + 1 разработчик | Согласование UX-решений, разбор пользовательских сценариев и review прототипов до передачи в разработку. Снижает количество переделок на UI-уровне. |
| -------- |-------- |---------- | ------------ |
| Tech Debt Review (30 ми) | Раз в месяц | Разработчики + PM | Аудит накопленного технического долга и выделение capacity на его погашение в следующем спринте. Критично для стартапа — MVP-код быстро становится legacy без контроля. |

**Hiring Plan** — команда под MVP EduFlex:**Development Framework** — выбор методологии:**Team Rituals** — расписание встреч команды:Задание полностью готово. Вот краткое резюме по каждому блоку:

**Hiring Plan.** Команда из 7 человек: PM, 2 fullstack-разработчика, 1 backend, 1 UX/UI, 1 DS (part-time) и 1 QA (part-time). Такой состав покрывает весь MVP без избыточности — DS и QA на part-time логичны, поскольку на старте их загрузка неравномерна.

**Development Framework.** Выбран Scrum с двухнедельными спринтами. Ключевые аргументы: быстрая валидация гипотез (требования от HR-клиентов будут уточняться после каждого пилота), прозрачность для инвесторов через velocity и демо, и возможность роста без смены методологии.

**Team Rituals.** Семь встреч с разной частотой — от ежедневного стендапа (15 мин) до ежемесячного Tech Debt Review. Особо важны Backlog Refinement (предотвращает сырые истории на планировании) и Product Sync (сокращает переделки на UI-уровне) — именно эти два ритуала чаще всего пропускают в стартапах и потом жалеют.
